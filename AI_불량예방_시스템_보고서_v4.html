2 현장 문제 재정의

컨베이어 이송 중 피도장 제품의 무게가 가벼울 경우 후크가 들리면서 톱니바퀴가 회전 유도장치에 제대로 맞물리지 못한다. 이로 인해 제품이 회전하지 않거나 불규칙하게 회전하여 도장 분포가 불균일해지고 품질 불량으로 이어진다.

현 상황: 제품 경량화 추세에 따라 발생 빈도가 꾸준히 증가. 발생 위치가 육안으로 식별 불가능하여 자동 라인 3기가 전량 수동 작업으로 전환된 상태. 생산성 저하 · 인건비 부담 · 품질 균일성 문제가 지속 발생.

NEW ROI 설계 방식 비교 분석 — 고정 ROI vs 동적 ROI

현재 설계(고정 ROI + Optical Flow + GRU)를 기반으로, 카메라는 고정이지만 톱니바퀴가 컨베이어를 따라 계속 전진하는 환경에서 "고정 ROI로 충분한가"를 재검토하고, YOLO 검출 기반 동적 ROI 방식과 구조적으로 비교 분석한다.

문제 제기: 현재 설계는 톱니바퀴가 회전 유도장치에 진입하는 특정 지점을 고정 ROI로 감시한다. 그러나 실제 현장에서는 컨베이어 전진 이동에 의해 객체 위치가 프레임 내에서 지속적으로 변화한다. 이 경우 고정 ROI 방식은 구조적 취약점을 가질 수 있으며, 검출/추적 기반 동적 ROI가 더 안정적일 가능성이 있다.

현재 설계 구조 — 고정 ROI + Optical Flow + GRU

현재 보고서는 아래 두 가지 가정을 전제로 고정 ROI 방식을 선택하였다.

전제 ①
문제 발생 지점 특정
불량은 주로 회전 유도장치 진입 지점에서 발생한다. 해당 구간만 집중 감시하면 충분하다는 설계 원칙.
전제 ②
판정 목적: 패턴 분석
"객체가 어디 있는가"보다 "픽셀 벡터가 정상적으로 흐르는가"가 핵심. 탐지보다 패턴 판정이 주 목적.

고정 ROI의 구조적 취약점 3가지

취약점 ①
관찰 시간 부족
고정 ROI를 통과하는 시간이 짧으면 GRU에 필요한 10~20프레임 시계열을 안정적으로 확보하기 어렵다. 공정 속도가 빠를수록 이 문제가 심화된다.
취약점 ②
Optical Flow 오염
톱니가 ROI 경계에 걸쳐 있을 때 픽셀 흐름에 회전 성분 + 평행이동 성분 + 잘림이 섞인다. GRU가 "회전 이상"이 아닌 "컨베이어 이동"을 학습할 위험.
취약점 ③
위치 편차에 취약
제품 크기 차이, 컨베이어 좌우 흔들림, 카메라 설치 오차가 누적되면 톱니가 고정 ROI를 벗어나는 경우가 발생한다. 이 경우 파이프라인 자체가 무효화된다.
현재 설계가 유효한 조건
① 진입 지점의 위치 편차가 매우 작고 일정할 때
② 공정 속도가 충분히 느려 ROI 내 체류 시간 동안 20프레임 이상 수집 가능할 때
③ 구현 단순성과 연산 경량화가 최우선 과제일 때

제안 방향 — YOLO 검출 기반 동적 ROI

YOLO를 전처리 앞단에 배치하여 매 프레임(또는 N프레임 간격)마다 톱니바퀴 위치를 자동 검출하고, 검출된 bounding box를 ROI로 활용한다. Optical Flow / GRU 판정 로직은 기존과 동일하게 유지된다.

핵심 아이디어: YOLO는 위치 찾기 담당, Optical Flow + GRU는 회전 패턴 판정 담당. 두 역할을 분리하여 각자의 강점을 살린다. YOLO가 "회전을 판정"하는 것이 아님.

동적 ROI 설계의 핵심 기술 과제

기술 과제 ①
이동 성분 분리 — 전진 vs 회전
컨베이어 전진 속도에 의한 글로벌 평행이동 벡터를 Optical Flow에서 제거하고, 톱니 로컬 좌표계 기준의 회전 벡터만 GRU에 입력해야 한다. 이를 위해 bbox 중심점 기준 affine alignment 또는 배경 흐름 추정 후 차감 방법을 검토.
기술 과제 ②
ID 일관성 유지 — 멀티 객체 추적
프레임 간 동일 톱니에 동일 ID가 부여되지 않으면 GRU의 20프레임 시계열이 서로 다른 객체 데이터로 오염된다. ByteTrack 또는 DeepSORT 기반 추적기로 ID 연속성을 보장해야 한다.
기술 과제 ③
YOLO 커스텀 데이터셋 — 학습 데이터 필요
톱니바퀴·후크·유도장치는 COCO의 일반 객체 클래스와 다르다. COCO pretrained 가중치를 초기값으로 활용하되, 현장 이미지 기반 커스텀 라벨링(gear / hook / guide)과 파인튜닝이 필수다. COCO 사전학습 자체는 기초 시각 특징 학습에 여전히 유효하다.
기술 과제 ④
bbox 겹침 처리 — NMS 임계값 튜닝
인접한 여러 톱니가 화면에서 겹쳐 보일 때 bbox가 붙거나 일부가 가려져 ROI가 오염될 수 있다. NMS(Non-Maximum Suppression) 임계값 튜닝과 ROI padding 크기 최적화가 필요하다.
YOLO 모델 설계 방향
커스텀 클래스 정의: gear (톱니바퀴) / hook (후크) / guide (회전 유도장치)
학습 전략: COCO pretrained YOLOv8 weights → 현장 이미지 라벨링 → fine-tuning
운영 전략: 최초 1회 YOLO 검출 → 이후 Tracker(KLT/ByteTrack)로 추적 → 미검출 N프레임 시 재검출 (연산 부하 분산)

3 센서 구성 확정

레이저 변위센서는 현장 물리 현상 재분석 결과 제외 결정. 최종 센서 구성은 비전카메라와 압력/깊이 센서 두 종류로 확정.

비전카메라 (Air Purge)
5MP급 고해상도
공정 속도가 느리므로 FPS를 낮추는 대신 해상도를 높여 기어 톱니의 미세한 진입 각도와 이탈 형상을 선명하게 포착.

Air Purge 하우징 필수 장착 — 분체 도장 환경의 분진 오염을 방지하여 카메라 신뢰성 유지.

ROI는 톱니바퀴가 회전 유도장치에 진입하는 지점으로 한정하여 배경 노이즈를 최소화.
압력/깊이 센서 (로드셀)
접촉력 시계열 수집
후크가 지지 가이드(Support Guide)에 가하는 접촉력을 시계열로 수집.

비전의 보조 신호로 활용하며, 카메라 오염 시에도 단독으로 Class 2(이탈 확정) 판정이 가능하도록 폴백 로직 구현.

정상 상태: 일정한 압력 파형 유지 / 후크 들림 시: 압력 감소 또는 충돌·스파이크 패턴 발생

4 모델 아키텍처 설계

비전 파이프라인과 압력 센서 파이프라인이 각각 독립적으로 특징을 추출한 뒤 융합 모듈에서 결합하는 멀티모달 구조로 설계. 비전 파이프라인은 ROI 설계 방식에 따라 두 가지 케이스로 분기된다.

비전 파이프라인 케이스 분기
Case A — 고정 ROI: 사전 지정된 진입 지점 ROI → Optical Flow → CNN → GRU. 현재 채택된 기본 설계.
Case B — YOLO 동적 ROI: YOLO 검출 → Tracker → 동적 ROI → Optical Flow → CNN → GRU. ROI 이탈 리스크가 클 때 전환 검토.
공통: 압력 센서 파이프라인(1D CNN)과 멀티모달 융합 구조는 두 케이스 모두 동일하게 적용.

Case A — 현재 채택 설계: 사람이 사전 지정한 고정 ROI 기반. YOLO 없이 전처리 → CNN → GRU 직행. 구현 단순, 연산 비용 최소. 진입 지점 편차가 작고 통과 시간이 충분할 때 최적.

비전 파이프라인 — 백본 / 넥 / 헤드
전처리 ROI 크롭 + Farneback Dense Optical Flow (H×W×2채널 속도 맵)
→
백본 (CNN) 귀납적 편향으로 소규모 데이터에서 안정적 학습
→
넥 (GRU) 10~20 프레임 단기 시계열, LSTM 대비 25% 적은 파라미터
→
헤드 Feature A 출력 (최종 분류)

Optical Flow 핵심 아이디어: "회전 중인가"라는 질문을 "픽셀 벡터가 일관되게 흐르는가"로 치환하여 톱니 구조의 반복성 문제를 구조적으로 회피. 정상 회전 시 벡터가 일정 방향·크기 유지, 회전 불량 시 벡터가 흐트러지거나 0에 수렴.

근거: Chen et al. (2025), Frontiers in Mechanical Engineering — Farneback Optical Flow 기반 모듈 @97.8% 달성

Case B — 전환 검토 설계: YOLO로 매 프레임 톱니 위치를 자동 검출하고 동적 ROI를 생성. Optical Flow / CNN / GRU 판정 로직은 Case A와 동일. 커스텀 학습 데이터와 Tracker 설계가 추가로 필요.

Case B 비전 파이프라인 — 4개 레이어 구조
L2. YOLO 검출 (신규) YOLOv8 커스텀 파인튜닝 — gear / hook / guide 클래스.
COCO pretrained 가중치 → 현장 이미지 fine-tuning.
출력: bbox (x, y, w, h) + confidence
→
L3. Tracker (신규) ByteTrack 또는 KLT 기반 ID 추적.
프레임 간 동일 객체 → 동일 ID 보장.
미검출 N프레임 → YOLO 재검출 트리거
→
L4. 동적 ROI (신규) bbox + padding → 로컬 ROI 크롭.
객체 중심 정렬 (전진 이동 성분 보정).
ID별 독립 ROI — 다중 톱니 병렬 처리
→
L5~7. Flow → CNN → GRU (공유) Case A와 동일한 Optical Flow + CNN + GRU.
단, ID별 독립 시퀀스 버퍼 유지.
각 ID 개별 Class 0/1/2 판정 후 집계

Case B 신규 추가 컴포넌트 상세

L2 · YOLO 검출 모듈
YOLOv8 커스텀 파인튜닝
역할: 매 프레임(또는 K프레임 간격)에서 톱니바퀴·후크 위치를 자동 검출해 bounding box 제공.

학습 전략:
① COCO pretrained YOLOv8 가중치로 초기화 (기초 시각 특징 — 엣지·윤곽·패턴 활용)
② 현장 촬영 이미지에 gear / hook / guide 클래스 직접 라벨링
③ 파인튜닝으로 산업 부품 특화 도메인 적응

운영 전략 (연산 최적화):
최초 1회 YOLO 검출 → 이후 Tracker(KLT)로 추적 → 미검출 N프레임 시 재검출.
매 프레임 YOLO 추론 대신 Tracker를 앞세워 Edge(Jetson Orin Nano) 연산 부하 분산.
L3 · Tracker 모듈
ByteTrack / KLT — 멀티 객체 ID 추적
역할: 프레임 간 동일 톱니에 동일 ID를 부여하여 GRU 시계열 버퍼의 데이터 혼입을 방지.

핵심 문제: YOLO는 프레임마다 독립적으로 박스를 출력하므로, 다음 프레임의 박스가 이전 프레임의 어느 객체인지를 별도로 매핑해야 한다. 이 매핑이 끊기면 ID-101의 20프레임 버퍼에 ID-102의 데이터가 섞여 GRU 판정이 오염된다.

ByteTrack 선택 이유: 검출 신뢰도가 낮은 박스도 추적에 활용하여 산업 현장의 분진·부분 가림 환경에서 ID 연속성 향상. SORT/DeepSORT 대비 ReID 네트워크 없이 경량 동작.
L4 · 동적 ROI 생성 모듈
bbox 기반 로컬 ROI — 이동 성분 분리
역할: Tracker가 제공한 ID별 bbox를 기반으로 Optical Flow 연산에 사용할 로컬 ROI를 생성.

Padding: bbox 그대로 쓰면 Optical Flow가 주변 맥락 정보를 놓칠 수 있음. 20~50px 패딩을 추가하여 회전 경계면 픽셀 벡터까지 포함.

이동 성분 보정 (핵심): 컨베이어 전진에 의한 글로벌 평행이동 벡터를 추정하고, 로컬 ROI 내 Optical Flow에서 해당 성분을 차감. 이후 GRU에는 순수 회전 성분만 입력됨.

보정 방법 후보: bbox 중심 좌표 프레임 간 차이로 평행이동 추정 → ROI Flow에서 차감 (간단), 또는 배경 영역 Flow 평균으로 글로벌 이동 추정 후 보정 (정밀).
L6 · ID별 시퀀스 버퍼
GRU 입력 — ID별 독립 슬라이딩 윈도우
Case A와의 차이: Case A는 고정 ROI에 대한 단일 시퀀스 버퍼를 운영하지만, Case B에서는 화면에 탐지된 톱니 N개마다 독립적인 20프레임 슬라이딩 윈도우를 유지한다.

버퍼 오염 방지: ID 재할당이 발생하면 해당 ID의 버퍼를 즉시 초기화(flush)하여 이전 객체 데이터 혼입 방지.

메모리: 최대 동시 추적 톱니 수 × 20프레임 × Flow 맵 크기(H×W×2). Edge 메모리(8GB) 내에서 관리 가능한 N 설정 필요.

Case B 구현 전제조건 체크리스트:
① 현장 이미지 기반 gear / hook / guide 라벨 데이터셋 구축 (최소 수백 장 이상)
② YOLO 파인튜닝 및 현장 정확도 검증 (분진 환경, 조명 변화 대응)
③ ByteTrack ID 연속성 현장 테스트 (컨베이어 속도별 ID 유지율 측정)
④ 이동 성분 보정 알고리즘 검증 (보정 전후 GRU 분류 성능 비교)
⑤ Edge(Jetson Orin Nano 8G) TensorRT FP16 변환 후 레이턴시 실측 (0.5초 인터락 기준 충족 여부)

전환 판단 흐름도

두 케이스의 점진적 전환 전략: Optical Flow → CNN → GRU 로직은 두 케이스가 완전히 공유한다. 따라서 Case A로 개발을 시작한 뒤, 현장 실측 결과에 따라 전처리 앞단(ROI 생성 모듈)만 교체하는 방식으로 Case B로 전환 가능하다. 전체 아키텍처를 다시 설계할 필요 없이 YOLO + Tracker 레이어만 앞에 추가하면 된다.

5 아키텍처 선택 비교 및 근거

데이터 효율, 추론 속도, 입력 데이터 궁합, 구현 복잡도 4개 기준으로 주요 아키텍처를 평가하였다.

아키텍처 데이터 효율 추론 속도 입력 궁합 구현 복잡도 채택 여부
CNN (백본) 높음 빠름 매우 적합 낮음 채택
GRU (넥) 높음 빠름 적합 낮음 채택
RNN 낮음 빠름 적합 낮음 미채택
LSTM 중간 중간 적합 중간 미채택
ViT 낮음 중간 적합 높음 2단계 검토
Video Swin 매우 낮음 느림 매우 적합 매우 높음 3단계 목표
CNN vs ViT 교차점 — 원저 논문 근거
Dosovitskiy et al. (2021), ICLR 2021 — ResNet(CNN)은 소규모에서 ViT보다 높은 성능, ViT는 14M~300M 규모에서 역전됨. "합성곱의 귀납적 편향은 소규모에서 유용하지만 대규모 학습이 이를 압도한다." 본 프로젝트 초기 이상 데이터(수백~수천 건)는 교차점(약 10M장)에 크게 못 미침.

6 학습 전략 — 반지도학습 구조

이상 클래스 데이터의 초기 부족 문제를 해결하기 위해, 정상 데이터를 먼저 대량 활용하는 반지도학습 구조를 채택.

1단계 착수~개발 초기
비지도 이상 탐지
CNN 오토인코더로 정상 Optical Flow 맵 복원을 학습. 운영 중 복원 오류가 임계값 초과 시 이상 판정. 라벨 없이 즉시 현장 적용 가능. 손실 함수: Reconstruction Loss (MSE)
2단계 개발~실증
점진적 라벨 누적 및 분류 모델
이상 이벤트 발생 시마다 라벨 부여 — 지도학습 데이터 축적. 1단계 오토인코더 인코더 가중치를 초기값으로 전이. Focal Loss 적용으로 클래스 불균형(정상 >> 이상) 대응. Class 0/1/2 분류 모델로 고도화.
3단계 데이터 1만 건 이상
Transformer 기반 고도화
Video Swin Transformer 또는 Cross-modal Attention 파인튜닝. 2단계 학습 가중치를 초기값으로 활용 — 처음부터 학습 대비 데이터 효율 극대화.

7 Edge-Server-HMI 아키텍처 설계

추론은 현장 Edge 디바이스에서 전담하여 0.5초 이내 PLC 인터락을 보장하고, 서버는 데이터 적재 및 MLOps 재학습에 집중하며, 운영자는 웹 대시보드를 통해 PC·태블릿으로 실시간 모니터링한다.
