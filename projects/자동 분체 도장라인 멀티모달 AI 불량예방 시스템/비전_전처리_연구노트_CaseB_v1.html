배경 연구 배경 및 범위

Case A(고정 ROI) 연구노트에서 전처리 설계는 "단일 고정 ROI → Farneback Dense Optical Flow → CNN → GRU" 구조로 확정되었다. 이 설계는 회전 유도장치가 고정 구조물이라는 전제 위에 성립한다.

그러나 현장 실측 결과에 따라 다음 두 상황이 발생하면 고정 ROI가 무력화된다:

한계 상황 ①
컨베이어 위치 편차
제품이 항상 동일한 위치로 진입하지 않아 톱니가 고정 ROI를 벗어나는 프레임 발생. ROI 밖에서 이탈이 시작되면 감지 불가.
한계 상황 ②
다중 톱니 동시 추적 필요
단일 ROI 버퍼로는 어느 톱니에서 이상이 발생했는지 특정 불가. N개의 톱니를 병렬로 독립 추적해야 하는 요구사항 발생.
Case B 해법
YOLO로 위치를 찾는다
YOLO가 매 프레임 톱니 위치를 자동 검출해 ROI를 동적으로 생성. Optical Flow + GRU 판정 로직은 Case A와 동일하게 유지.

핵심 역할 분리: YOLO는 위치 찾기 담당, Optical Flow + GRU는 회전 패턴 판정 담당. 두 역할을 분리하여 각자의 강점을 살린다. YOLO가 "회전을 판정"하는 것이 아님.

이 문서는 Case B 전환 시 맞닥뜨릴 다섯 가지 설계 의문을 검토하고 결론을 도출하는 과정을 기록한다. 각 질문은 실제 고민의 흐름 순서대로 정리되었으며, 의문 → 검토 → 결론의 구조를 따른다.

Q1 YOLO를 매 프레임 추론해야 하는가 — 연산 부하

최초 의문
YOLOv8을 매 프레임 추론하면 Jetson Orin Nano 8G에서 0.5초 인터락 기준을 맞출 수 있는가? 실시간 처리를 위해 매 프레임 YOLO 추론이 반드시 필요한가?

오개념 — 매 프레임 YOLO 추론이 필수라는 생각

오개념
동적 ROI이므로 ROI 위치를 매 프레임 새로 찾아야 한다 → YOLO를 매 프레임 돌려야 한다 → Edge 연산 한계 초과 → 실시간 처리 불가.

왜 오개념인가

YOLO가 "위치를 한 번 찾으면" 이후 프레임은 훨씬 가벼운 Tracker(KLT 또는 ByteTrack)로 추적할 수 있다. Tracker는 이전 프레임의 픽셀 특징점을 다음 프레임에서 추적하는 방식으로, YOLO 추론 없이도 bbox를 업데이트한다.

YOLO 역할 (최초 + 재검출)
위치 확보 담당
최초 1회 또는 Tracker가 ID를 잃었을 때만 YOLO 추론 실행. bbox (x, y, w, h) + 클래스(gear/hook/guide) 출력. 무거운 추론을 최소 횟수로 제한.
Tracker 역할 (매 프레임)
위치 유지 담당
KLT Optical Flow 기반으로 특징점을 추적, 매 프레임 bbox를 갱신. YOLO 추론 없이 경량 연산만으로 동일 객체의 위치 유지.

운영 전략 — YOLO 재검출 트리거

결론
매 프레임 YOLO 추론이 아닌 "Tracker 선행 + 재검출 트리거" 구조로 설계.
YOLO는 "감시 카메라"가 아니라 초기 위치 확보 + 이탈 시 복구 역할만 담당한다. TensorRT FP16 변환 후 레이턴시 실측 필수 — 0.5초 인터락 기준 충족 여부 확인.

Q2 Tracker가 왜 필수인가 — ID 혼입 문제

최초 의문
YOLO가 매 프레임(또는 K프레임마다) bbox를 주면 그것을 그냥 ROI로 쓰면 되지 않는가? Tracker를 별도 레이어로 두는 이유가 무엇인가?

문제의 구조 — YOLO는 프레임마다 독립 출력한다

YOLO는 프레임마다 독립적으로 박스를 출력한다. t=0에서 출력된 bbox가 t=1에서 출력된 어느 bbox와 "같은 톱니"를 의미하는지 YOLO는 알려주지 않는다.

Tracker 없을 때
ID 매핑 단절 → GRU 버퍼 오염
t=0의 bbox A가 t=1의 bbox B와 같은 톱니인지 보장 없음. GRU는 ID-101의 20프레임 시퀀스 중간에 ID-102의 Flow 맵이 섞이게 됨 → 판정 오염.
Tracker 있을 때
프레임 간 동일 객체에 동일 ID 보장
t=0에 부여된 ID-101이 t=1, t=2 … 에서도 동일한 톱니를 가리킴. GRU의 20프레임 시퀀스 버퍼가 항상 같은 객체 데이터만 포함.

ByteTrack 선택 근거

ByteTrack
분진 환경에서 ID 연속성 강화
선택 이유: 검출 신뢰도가 낮은 박스도 추적에 활용하여 산업 현장의 분진·부분 가림 환경에서 ID 연속성 향상.

SORT/DeepSORT 대비 장점: ReID 네트워크 없이 경량 동작. Edge(Jetson Orin Nano) 메모리·연산 부담 최소화.

KLT와의 조합: 최초 검출 후 KLT(Lucas-Kanade Optical Flow 기반 특징점 추적)로 frame-by-frame 추적. KLT가 실패하면 ByteTrack ID 매핑으로 복구.
결론
Tracker는 "편의 기능"이 아니라 GRU 시계열 무결성을 보장하기 위한 구조적 필수 요소다.
Tracker 없이 YOLO bbox를 직접 ROI로 쓰면 GRU의 20프레임 버퍼가 서로 다른 객체 데이터로 오염되어 회전 패턴 판정이 불가능해진다.

Q3 동적 ROI 생성 — bbox를 그대로 쓰면 되는가

최초 의문
Tracker가 주는 ID별 bbox 좌표를 그대로 Optical Flow 입력 영역으로 크롭하면 되는 것 아닌가? 추가 처리가 왜 필요한가?

두 가지 이유로 bbox를 그대로 쓸 수 없다

문제 ① Padding 부재
경계면 Flow 벡터 손실
bbox 경계를 딱 맞춰 크롭하면 Optical Flow가 경계면 픽셀 벡터를 놓친다. 톱니 이빨이 맞물리는 접촉면 벡터가 핵심 회전 신호인데, 이 부분이 bbox 가장자리에서 잘려나갈 수 있다.
문제 ② 이동 성분 중첩
컨베이어 전진 벡터가 섞인다
ROI가 톱니를 따라 이동하므로, ROI 내부 Optical Flow에는 컨베이어 전진에 의한 글로벌 평행이동 벡터가 중첩된다. GRU는 순수 회전 성분만 받아야 올바른 판정이 가능하다.

문제 ① 해법 — Padding 추가

bbox에 20~50px 패딩을 추가하여 회전 경계면 픽셀 벡터까지 포함한다. 패딩 크기는 현장에서 톱니 이빨 크기 대비 적절한 여유를 측정해 결정.

문제 ② 해법 — 이동 성분 분리 (핵심 설계 과제)

Case A와의 근본적 차이: Case A에서는 고정 ROI 덕분에 컨베이어 이동이 "톱니가 ROI를 통과하는 현상"으로 자연스럽게 처리되었다. Case B에서는 ROI가 톱니를 따라다니므로 이동 성분을 명시적으로 제거해야 한다.

보정 방법 비교

방법 원리 장점 단점 추천 상황
방법 A
bbox 중심 차분		 bbox 중심 좌표의 프레임 간 차이로 평행이동 벡터 추정 → ROI Flow에서 차감 구현 간단, 연산 경량 Tracker 오차가 보정에 직접 전파 컨베이어 속도 일정, 편차 작을 때
방법 B
배경 Flow 평균		 ROI 외부 배경 영역 Flow 평균으로 글로벌 이동 벡터 추정 후 보정 Tracker 오차에 독립적, 정밀 배경 영역 설정 추가 필요, 분진 환경에서 노이즈 컨베이어 속도 가변, 정밀도 요구 시
결론
동적 ROI = bbox + 20~50px padding + 이동 성분 보정.
padding은 회전 경계면 신호 보존을 위해 필수. 이동 성분 보정은 두 방법 모두 구현 후 현장 GRU 분류 성능을 비교하여 최종 채택.

Q4 ID별 시퀀스 버퍼 — Case A와 무엇이 다른가

최초 의문
GRU 입력 버퍼 구조가 Case A와 동일한가? Case B는 동시에 여러 톱니를 추적하는데, 버퍼를 어떻게 관리해야 하는가?

Case A의 버퍼 구조 — 단일 슬라이딩 윈도우

Case A는 고정 ROI에 대한 단일 슬라이딩 윈도우 1개만 운영했다. 톱니가 순차로 ROI를 통과하므로 하나의 버퍼로 충분했다.

Case B의 버퍼 구조 — ID별 독립 슬라이딩 윈도우

Case B에서는 화면에 추적 중인 톱니 N개마다 독립적인 20프레임 슬라이딩 윈도우를 유지해야 한다. GRU는 각 ID의 버퍼로부터 독립적으로 Class 0/1/2를 판정하고 결과를 집계한다.

버퍼 관리 핵심 규칙

규칙 ①
ID 재할당 시 버퍼 즉시 초기화(flush)
ID 재할당이 발생하면 해당 ID의 버퍼를 즉시 초기화. 이전 객체의 Flow 데이터가 새 객체의 판정에 영향을 주지 않도록 보호.
규칙 ②
최대 동시 추적 수 N 설정
메모리 상한: N × 20프레임 × Flow 맵(H×W×2). Edge(8GB) 내에서 관리 가능한 N 결정 필요. N은 현장 동시 진입 톱니 수 기준으로 설정.
규칙 ③
판정 기준 — 버퍼 채워진 ID만
20프레임이 채워지지 않은 ID는 판정 보류. 부분 시퀀스로 GRU 판정 시 오탐(False Alarm) 발생 가능. 충분한 시퀀스 확보 후 판정.
결론
버퍼 구조는 Case A 대비 N배로 확장.
단일 버퍼 1개 → ID별 독립 버퍼 N개. ID 재할당 시 즉시 flush, 20프레임 미만 ID는 판정 보류. N 상한은 현장 동시 진입 톱니 수와 Edge 메모리를 고려해 결정.

Q5 YOLO 커스텀 학습 — 무엇을 준비해야 하는가

최초 의문
YOLOv8 COCO pretrained 모델을 그냥 쓰면 안 되는가? 어떤 추가 준비가 필요하고, 분진 환경은 어떻게 대응하는가?

오개념 — COCO pretrained 그대로 사용 가능하다는 생각

오개념
COCO pretrained YOLOv8은 80개 클래스를 검출한다. 산업 부품도 그 중 하나로 포함되어 있을 것이다 → 그냥 쓰면 된다.

왜 오개념인가

COCO 80개 클래스에는 gear(톱니바퀴), hook(후크), guide(유도장치) 등 산업 특화 부품이 포함되어 있지 않다. COCO pretrained 가중치를 그대로 쓰면 이 객체들이 검출되지 않거나 엉뚱한 클래스로 분류된다.

pretrained 가중치의 올바른 활용법: COCO pretrained 가중치를 초기화값(기초 시각 특징 — 엣지·윤곽·패턴)으로만 활용하고, 현장 이미지 기반 파인튜닝(Fine-tuning)으로 도메인 적응(Domain Adaptation)을 수행한다.

학습 전략

YOLOv8 커스텀 파인튜닝 절차
3단계 — 초기화 → 라벨링 → 파인튜닝
Step 1. 초기화: COCO pretrained YOLOv8 가중치로 초기화. 저수준 시각 특징(엣지, 윤곽, 텍스처) 재사용으로 학습 효율 향상.

Step 2. 데이터셋 구축: 현장 촬영 이미지에 아래 3개 클래스 직접 라벨링. 최소 수백 장 이상 확보 권장.
   · gear — 톱니바퀴 (회전 주체)
   · hook — 후크 암 (이탈 감지 대상)
   · guide — 회전 유도장치 (고정 기준점)

Step 3. 파인튜닝: 분진·조명 변화 대응 Augmentation 포함. 현장 정확도 목표: AP@0.5 ≥ 0.85 (분진 환경 기준).

분진 환경 대응

문제
분체 도장 환경의 특수성
분진 부유, 부분 가림(Occlusion), 조명 불균일. 일반 제조 환경 대비 검출 신뢰도 저하 요인 다수.
대응 — 데이터 Augmentation
환경 변화 시뮬레이션
학습 시 Gaussian noise, blur, 밝기 변동, 랜덤 가림(Cutout) 적용. 분진 환경을 가상으로 재현해 모델 강건성 향상.
대응 — ByteTrack 설계
낮은 신뢰도 bbox도 추적 활용
ByteTrack이 낮은 confidence bbox도 추적에 활용하는 이유가 바로 이 분진 환경. 일시적 검출 실패 시에도 ID 연속성 유지.
결론
커스텀 데이터셋 구축이 Case B 전환의 핵심 선행 조건.
COCO pretrained 가중치는 초기화값으로만 활용하고, 현장 이미지 gear/hook/guide 라벨링 + 분진 환경 Augmentation 포함 파인튜닝이 필수. 최소 수백 장 이상 라벨 데이터 확보 후 AP@0.5 기준 현장 정확도 검증.

결론 전처리 파이프라인 확정안 (Case B)

다섯 가지 질문의 검토를 종합한 Case B 비전 전처리 파이프라인 확정안이다.

레이어 구성

확정
YOLO 검출 → Tracker → 동적 ROI(padding + 이동 성분 보정) → Optical Flow → ID별 버퍼 → CNN → GRU
L1. 카메라: 5MP 고해상도, Air Purge 하우징. Case A와 동일.

L2. YOLO v8 ★ 신규: 커스텀 파인튜닝(gear/hook/guide). 최초 1회 검출 + 재검출 트리거 시에만 실행. TensorRT FP16 변환으로 Edge 최적화.

L3. Tracker (ByteTrack) ★ 신규: ID 부여, 프레임 간 동일 객체 연속성 보장. 매 프레임 경량 추적. KLT 특징점 추적 조합.

L4. 동적 ROI 생성 ★ 신규: bbox + 20~50px padding. 이동 성분 보정(방법 A/B 비교 후 채택). ID별 독립 로컬 크롭.

L5. Farneback Dense Optical Flow 공유: H×W×2채널 속도 맵 생성. 순수 회전 성분만 포함(이동 성분 보정 후). Case A와 동일 알고리즘.

L6. ID별 독립 슬라이딩 윈도우 ★ 신규: N개 버퍼 병렬 운영. 각 버퍼 20프레임. ID 재할당 시 즉시 flush.

L7~. CNN → GRU → Class 0/1/2 공유: Case A와 동일. ID별 독립 판정 후 결과 집계.

비교 Case A vs Case B — 전처리 구조 비교

점진적 전환 전략
Optical Flow → CNN → GRU 로직은 두 케이스가 완전히 공유한다. 따라서 Case A로 개발을 시작한 뒤, 현장 실측 결과에 따라 전처리 앞단(ROI 생성 모듈)만 교체하는 방식으로 Case B로 전환 가능하다. 전체 아키텍처를 다시 설계할 필요 없이 YOLO + Tracker 레이어만 앞에 추가하면 된다.
항목 Case A — 고정 ROI Case B — YOLO 동적 ROI
ROI 결정 방식 1회 수동 지정, 이후 고정 매 프레임 YOLO → Tracker → 자동 생성
이동 성분 보정 불필요 (구조적으로 자동 처리) 명시적 보정 필수
시퀀스 버퍼 단일 버퍼 1개 ID별 독립 버퍼 N개
다중 톱니 동시 처리 불가 (단일 ROI) 가능 (병렬 처리)
위치 편차 대응 취약 (ROI 이탈 시 감지 불가) 강건 (YOLO가 항상 위치 추적)
구현 복잡도 낮음 높음 (YOLO 학습 + Tracker 추가)
추가 학습 데이터 Flow 학습 데이터만 + YOLO 커스텀 라벨 데이터셋
Edge 연산 부하 최소 중간 (Tracker 선행으로 최적화)
Optical Flow → CNN → GRU ← 완전 공유 (동일) →

언제 Case B로 전환하는가: ① 현장 실측에서 제품 위치 편차가 고정 ROI를 벗어나는 빈도가 높을 때, ② 다중 톱니 동시 독립 추적이 요구될 때. 둘 중 하나라도 해당하면 Case B 전환 검토.

체크 구현 전제조건 체크리스트

Case B 전환 전 반드시 완료해야 할 항목들이다. 모든 항목이 충족된 후 Case B 파이프라인을 실제 운영에 투입한다.

  • ① 커스텀 라벨 데이터셋 구축 현장 이미지 gear / hook / guide 클래스 라벨링. 분진 환경 포함 최소 수백 장 이상. 분진·조명 변화 Augmentation 적용.
  • ② YOLOv8 파인튜닝 및 현장 정확도 검증 분진 환경·부분 가림 대응 검증. 목표: AP@0.5 ≥ 0.85 (분진 환경 기준). 오검출(False Detection) 빈도 확인.
  • ③ ByteTrack ID 연속성 현장 테스트 컨베이어 속도별 ID 유지율 측정. 분진·가림 환경에서 ID 단절 빈도 확인. 허용 기준 설정.
  • ④ 이동 성분 보정 알고리즘 검증 방법 A(bbox 중심 차분) vs 방법 B(배경 Flow 평균) 모두 구현. 보정 전후 GRU 분류 성능(Accuracy, F1) 비교 후 채택.
  • ⑤ Edge 레이턴시 실측 Jetson Orin Nano 8G에서 TensorRT FP16 변환 후 전체 파이프라인 레이턴시 측정. 0.5초 인터락 기준 충족 여부 확인.
  • ⑥ N(최대 동시 추적 수) 결정 현장 동시 진입 톱니 수 실측. N × 20프레임 × Flow 맵 크기 메모리 추산. Edge 8GB 내 관리 가능한 N 상한 확정.
