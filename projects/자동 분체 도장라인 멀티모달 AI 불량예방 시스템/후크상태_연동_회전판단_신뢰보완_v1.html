1 검토 배경 및 문제 제기

기존 설계에서 카메라는 톱니바퀴 회전 상태 감지에 집중하도록 구성되어 있다. 그러나 분체 도장 라인의 실제 운용 구조를 보면, 후크에 도장 대상이 걸려 있지 않은 상태에서도 톱니는 컨베이어 벨트에 의해 이송되며 회전 유도장치를 통과한다. 이 경우 카메라 2가 톱니의 정지 상태를 감지하면 회전 불량으로 오판정할 위험이 구조적으로 존재한다.

문제 상황
도장 대상이 걸리지 않은 빈 후크의 톱니가 회전하지 않는 것은 정상 상태임에도 불구하고, 카메라 2가 이를 단독으로 판정할 경우 "회전 불량"으로 잘못 분류할 수 있다. 이는 오경보(False Alarm)를 발생시키고 판정 신뢰도를 훼손한다.
핵심 검토 질문
카메라 1이 판단한 "후크 적재 여부"를 카메라 2의 회전 판정에 어떻게 연동하면, 빈 후크로 인한 오경보 없이 실제 회전 불량만을 정확히 감지할 수 있는가?
검토 결론 (선요약)
구조적으로 완전히 가능하며, 두 카메라가 서로 다른 객체를 독립적으로 탐지하되 판정 단계에서 상태값을 결합하는 방식이 가장 안정적이다. 카메라 1은 "후크에 도장 대상이 걸려 있는가"를 판단하고, 카메라 2는 "톱니가 정상 회전하고 있는가"를 판단한다. 두 판단값을 컨베이어 속도 기반 큐(Queue)로 연결하면, 빈 후크 상태에서 카메라 2의 회전 판정 결과를 무시(스킵)하거나 아예 추론을 건너뛸 수 있다.

2 카메라별 역할 분리 설계

두 카메라는 설치 위치, 감지 대상, 출력 구조가 완전히 분리된다. 각자의 모델(YOLO 또는 분류기)은 독립적으로 학습되고, 공유하는 것은 판정 단계에서 사용하는 상태값 하나뿐이다.

카메라 1
후크 구간 — 적재 상태 판단
감지 대상: 후크 본체, 도장 대상물(피도장체)
판정 내용: 후크에 도장 대상이 걸려 있는가 (LOADED / EMPTY)
출력값: 후크 ID + 상태 + 감지 타임스탬프
탐지 방식: YOLO 커스텀 모델 (hook, object 2-class)
카메라 2
회전 유도 구간 — 톱니 회전 판단
감지 대상: 톱니(기어) 부위
판정 내용: 톱니가 정속으로 정상 회전하고 있는가 (OK / FAIL)
출력값: 회전 상태 + 판정 타임스탬프
탐지 방식: Optical Flow → GRU 분류기 (기존 파이프라인 유지)
항목 카메라 1 카메라 2
설치 위치 후크 이송 구간 (회전 유도장치 진입 전) 회전 유도장치 구간 (톱니 맞물림 지점)
주요 감지 객체 후크, 도장 대상물 톱니(기어)
모델 구조 YOLO 검출 (2-class: hook / object) Optical Flow + GRU (기존 유지)
출력 상태값 LOADED / EMPTY OK / FAIL
판정 독립성 완전 독립 (단독 판정) 카메라 1 상태값 참조 후 최종 판정
다른 카메라 의존도 없음 카메라 1 결과를 보조 조건으로 사용

서로 다른 객체를 탐지하는 것이 가능한가: 완전히 가능하다. 두 카메라는 각각 독립된 YOLO/Flow 모델을 구동하며, 물리적 설치 위치도 다르다. 공유되는 것은 상태값(LOADED/EMPTY)을 담는 소프트웨어 큐 하나뿐이다. 이는 기존 분산 비전 시스템에서 흔히 사용되는 구조다.

3 상태 전달 메커니즘 — 후크 상태 큐

카메라 1이 생성한 후크 상태값을 카메라 2가 사용하려면, 두 카메라 사이의 물리적 거리 + 컨베이어 이동 시간을 고려한 동기화가 필요하다. 이를 위해 "후크 상태 큐(Hook State Queue)"를 중간 레이어로 운용한다.

동작 원리

큐 항목 구조 및 동작 흐름

# 카메라 1: 후크 감지 시 큐에 push hook_queue.put({ "hook_id": generate_id(), # 후크 고유 식별자 "status": "LOADED" | "EMPTY", # 도장 대상 적재 여부 "detected_at": time.time(), # 감지 타임스탬프 "eta_cam2": time.time() + d / v, # 예측 도달 시간 (d: 거리, v: 속도) }) # 카메라 2: 톱니 감지 시 큐에서 매칭 pop def find_matching_hook(queue, now, tolerance=0.5): return min(queue, key=lambda h: abs(h["eta_cam2"] - now), default=None)

엔코더 없이 속도 추정: 컨베이어 엔코더 연동이 어려운 경우, 카메라 1이 연속으로 감지하는 동일 후크의 프레임 간 위치 이동량(bbox 중심 차분)으로 속도를 간접 산출할 수 있다. 단, 정밀도가 낮으므로 허용 오차(tolerance) 범위를 넉넉히 설정해야 한다.

4 최종 판정 로직 설계

카메라 1의 후크 상태값(LOADED / EMPTY)과 카메라 2의 회전 판정값(OK / FAIL)을 조합하면 4가지 경우의 수가 생긴다. 이 조합이 최종 판정의 전부다.

케이스 A · 정상
LOADED + OK → PASS
도장 대상 있음 + 톱니 정상 회전. 정상 통과. 별도 조치 불필요.
케이스 B · 경보
LOADED + FAIL → ALARM
도장 대상 있음 + 회전 불량. 핵심 감지 대상. 즉시 인터락 신호 발생.
케이스 C · 무시
EMPTY + FAIL → SKIP
빈 후크 + 회전 없음. 이는 정상이다. 카메라 2 판정 결과를 무시하고 로그만 기록.
케이스 D · 데이터 검토
EMPTY + OK → LOG
빈 후크인데 회전이 감지됨. 발생 빈도가 낮아야 정상. 높으면 카메라 1 오탐 가능성 검토.
def final_judgment(hook_status, rotation_ok): if hook_status == "LOADED" and rotation_ok: return "PASS" # 정상 elif hook_status == "LOADED" and not rotation_ok: return "ALARM" # 회전 불량 — 인터락 신호 발생 elif hook_status == "EMPTY": log_skip(rotation_ok) # 회전 결과 무관하게 스킵 (로그만) return "SKIP"
판정 설계 핵심 원칙
회전 판정은 LOADED 상태에서만 유효하다
빈 후크(EMPTY)에서 회전 불량이 감지되는 것은 물리적으로 당연한 현상이며 불량이 아니다. 이를 무시하지 않으면 오경보율이 급증하고 운용자 신뢰도가 떨어진다. 따라서 후크 상태값은 단순한 참고 데이터가 아니라, 회전 판정의 전제 조건(Gate)으로 설계해야 한다.

5 구현 방법 비교 — 방법 A vs 방법 B

카메라 2가 빈 후크의 회전 판정을 무시하는 방식은 두 가지로 구현할 수 있다. 두 방법은 "언제 무시할 것인가"의 타이밍이 다르다.

방법 A (권장) — 판정 후 무시
탐지는 하되, 판정 단계에서 조건 분기
카메라 2는 항상 톱니를 감지하고 회전 상태를 판정한다. 단, 최종 판정 함수에서 후크 상태값이 EMPTY이면 결과를 무시(SKIP)하고 로그만 남긴다.

장점: 구조 단순, 빈 후크 회전 데이터도 별도 수집 가능, 타이밍 오류 시 안전 측에서 동작
단점: GPU 추론이 항상 수행됨 (연산 낭비 소폭 발생)
방법 B — 추론 자체 스킵
프레임 수신 시 큐 선조회, EMPTY면 추론 생략
카메라 2가 프레임을 받기 전에 큐를 조회하여 해당 후크가 EMPTY이면 모델 추론 자체를 건너뛴다.

장점: Edge 연산 부하 절감 (특히 빈 후크 비율이 높은 라인에서 유리)
단점: 큐 조회 타이밍이 빠를 경우 LOADED 후크를 스킵하는 사고 위험. 타이밍 정밀도 요구 수준이 높음
비교 항목 방법 A (판정 후 무시) 방법 B (추론 스킵)
무시 타이밍 추론 완료 후 판정 함수에서 분기 프레임 수신 시 선조회 후 추론 전 분기
GPU 연산 항상 수행 LOADED 후크만 수행
오탐 안전성 높음 (추론 후 조건 분기이므로 안전) 중간 (타이밍 오류 시 LOADED 스킵 위험)
빈 후크 데이터 수집 가능 (로그 기록) 불가 (추론 미수행)
구현 복잡도 낮음 중간 (큐 선조회 타이밍 정밀 설계 필요)
권장 적용 조건 초기 도입 / 라인 속도 변동 큰 환경 Edge 리소스 제약 심하고 빈 후크 비율이 높은 환경

6 전체 판정 파이프라인

7 리스크 및 대응 방안

리스크 발생 조건 영향 대응 방안
큐 매칭 실패 컨베이어 속도 급변 또는 엔코더 미연동 후크 상태값이 틀린 후크에 매칭됨 허용 오차 창(tolerance window) 적절히 설정. 엔코더 연동 우선 검토.
카메라 1 오탐 (EMPTY → LOADED 오판) 분진, 역광, 부분 가림 LOADED 후크가 EMPTY로 판정 → 회전 불량 미감지 카메라 1 모델 정확도 목표 Recall ≥ 0.95 (LOADED 클래스). 분진 Augmentation 필수.
카메라 1 오탐 (LOADED → EMPTY 오판) 동일 환경 실제 불량이 SKIP 처리됨 — 미검출 위와 동일. LOADED → EMPTY 오판이 더 치명적이므로 Recall 최우선 최적화.
큐 TTL 초과 (항목 만료) 카메라 2 처리 지연이 예측보다 큰 경우 큐 항목이 만료되어 매칭 불가 TTL을 예측 도달 시간의 1.5배로 설정. 만료 시 기본값을 LOADED로 설정(안전 측).
후크 간격 불규칙 수동 걸기 작업으로 간격 편차 발생 큐 순서 불일치 타임스탬프 기반 매칭 사용. 순서 인덱스보다 시간 기반 ETA 매칭이 안정적.

안전 측 원칙: 큐 매칭에 실패하거나 TTL 초과로 후크 상태값을 확인할 수 없는 경우, 기본값을 LOADED(적재 있음)으로 처리한다. 이렇게 하면 불필요한 경보가 소폭 증가할 수 있으나, 실제 회전 불량을 미감지하는 치명적 실패는 방지된다.

8 구현 전제 조건 체크리스트

본 설계를 현장에 투입하기 전 반드시 완료해야 할 항목이다.

  • ① 카메라 1 설치 위치 확정 회전 유도장치 진입 전 구간에서 후크와 도장 대상물이 동시에 프레임 내에 포착되는 위치 실측. 최소 조도 조건 및 분진 환경 하 시인성 확인.
  • ② 카메라 1 YOLO 학습 데이터 구축 hook / object 2-class 라벨링. 빈 후크 / 적재 후크 / 부분 가림 / 분진 환경 포함. 현장 실촬영 이미지 최소 수백 장 이상 확보. Augmentation: 분진, 역광, 명암 변화.
  • ③ 카메라 1 YOLO 정확도 검증 LOADED 클래스 Recall ≥ 0.95 (미감지가 오경보보다 치명적). EMPTY 클래스 Precision ≥ 0.90. 분진 환경 별도 테스트셋으로 검증 필수.
  • ④ 컨베이어 엔코더 연동 또는 속도 추정 방식 확정 엔코더 신호 직접 연동이 최선. 불가 시 카메라 1의 bbox 이동량 기반 속도 추정 구현. 속도 추정 오차가 허용 오차 창(tolerance) 내에 들어오는지 현장 실측으로 검증.
